# ADR 0017: Module Schema Registry And Migration Ownership

Status: Proposed for task 3373

Date: 2026-06-25

## Context

Rusty Crew is intentionally keeping one service-owned database per instance.
Runtime coordination, dense profile memory, future roleplay lore, profile
registry state, and module-owned service data should not require separate
database services by default.

At the same time, module-owned data cannot mean ad hoc table creation. Rusty
Crew needs expansion space for roleplay/lore and other future modules without
letting schema drift leak out of the Rust persistence boundary or making a
future SQLite to PostgreSQL move harder.

The storage strategy design recommends:

- SQLite remains the embedded/local default.
- PostgreSQL remains a scale backend target.
- Rust owns schema, migrations, repositories, transaction boundaries, and
  backend capability reporting.
- TypeScript owns brain/tool integration, provider calls, adapter presentation,
  and module orchestration over typed APIs.
- Users and agents inspect data through diagnostics and curated read-only query
  catalogs, not raw SQL.

## Decision

Rusty Crew will add a Rust-owned module schema registry. A module that needs
durable service data registers a schema bundle. The service validates those
bundles, applies migrations, reports diagnostics, and exposes typed repository
or query APIs.

Schema bundles are compiled Rust descriptors for v1. A manifest/codegen layer
may be added later, but the first production boundary should be regular Rust
types and tests so migrations, backend capabilities, and naming validation live
with `core-persistence`.

TypeScript modules may not create tables, run migrations, open database
connections, or issue raw SQL. They may call explicit bridge/service APIs, use
read-only catalog queries, and provide module logic above those APIs.

## Schema Bundle Shape

A module schema bundle contains:

- `module_id`: stable snake_case identifier, e.g. `roleplay_lore`.
- `schema_version`: module-local monotonically increasing integer.
- `owner`: Rust crate/module that owns migrations and repositories.
- `logical_stores`: named stores exposed to diagnostics and query catalogs.
- `tables`: physical table declarations or references to migration fragments.
- `indexes`: declared indexes and query-plan expectations.
- `retention`: archive/purge/compact rules, or explicit `manual_only`.
- `capability_requirements`: backend features needed by the bundle.
- `repository_contracts`: Rust APIs that are the supported access path.
- `query_catalog_entries`: optional read-only diagnostic queries.
- `export_hooks`: stable export format for backup/portability.
- `import_hooks`: validation and import/restore behavior.
- `migration_notes`: operator-facing notes for irreversible or expensive steps.

The registry also records runtime state:

- installed module version;
- latest known descriptor fingerprint;
- migration status;
- capability status for the active backend;
- diagnostics such as row counts, estimated size, and retention health.

## Namespace Rules

Module ids use lowercase ASCII snake_case:

- allowed: `roleplay_lore`, `memory_spaces`, `simple_kv`;
- disallowed: mixed case, hyphenated ids, SQL keywords, dynamic user input.

Physical names are generated by the registry:

- tables: `module_<module_id>_<table_name>`;
- indexes: `idx_module_<module_id>_<table_name>_<purpose>`;
- triggers, if needed: `trg_module_<module_id>_<table_name>_<purpose>`;
- FTS/search artifacts: `module_<module_id>_<store_name>_search`.

Table and store names also use snake_case and are checked against a closed
identifier validator. Modules do not provide raw physical names.

Core runtime tables do not use the `module_` prefix. Module bundles may not
declare tables outside their namespace. Cross-module foreign keys are
disallowed in v1 unless a later ADR defines shared ownership.

## Lifecycle

1. Compile descriptors into the Rust persistence crate or a Rust storage-module
   crate consumed by it.
2. At service startup, collect registered bundles from a closed registry.
3. Validate descriptor shape, namespace safety, version monotonicity, and
   backend capability requirements.
4. Compare installed module versions against descriptors.
5. If compatible, apply module migrations inside service-owned migration
   orchestration.
6. If incompatible or unsafe, block startup or mark the module unavailable
   depending on severity.
7. Expose registry diagnostics through admin/read-only APIs.
8. Route all runtime access through typed repositories and query catalog
   entries.
9. Run retention only through module-declared maintenance hooks.
10. Export/import via module hooks, not direct file/table copying.

Rollback is not implicit. If a migration can be reversed, the descriptor may
declare a rollback hook for development/test use. Production rollback defaults
to blocked with an operator-facing message and a backup/restore requirement.

## Backend Capability Gates

Bundles declare portable requirements rather than backend names when possible:

- `transactions`;
- `json_documents`;
- `full_text_search`;
- `case_insensitive_search`;
- `concurrent_writers`;
- `generated_columns`;
- `online_index_build`;
- `advisory_locks`.

Backend-specific fragments are allowed only behind capability gates. For
example, a lore search store may provide:

- SQLite FTS5 migration fragment when `full_text_search=sqlite_fts5`;
- PostgreSQL `tsvector`/GIN migration fragment when
  `full_text_search=postgres_tsvector`.

Repository and service APIs must stay capability-shaped. A frontend or TS brain
tool should learn that lore search is degraded or unavailable, not that it must
send different SQL because the backend is SQLite.

## TypeScript Boundary

TypeScript may:

- call typed repository/bridge APIs;
- expose agent-facing tools over those APIs;
- compose frontend/admin API responses;
- run provider calls for capture, summarization, retrieval decisions, and
  prompt assembly;
- request module diagnostics/query catalog reads;
- define UI/plugin-level preferences that Rust validates before persistence.

TypeScript may not:

- create, alter, or drop tables;
- own schema migration order;
- open the Crew DB directly;
- issue raw SQL through the bridge;
- register new physical table names at runtime;
- bypass Rust repository invariants for revisions, retention, or provenance.

This keeps module business logic flexible without turning TypeScript into a
second migration system.

## Example: Roleplay Lore Module

Bundle:

- `module_id`: `roleplay_lore`
- `schema_version`: `1`
- stores:
  - `worlds`
  - `entities`
  - `lore_entries`
  - `relationships`
  - `timelines`
  - `provenance_events`
- capabilities:
  - required: `transactions`, `json_documents`
  - optional/degraded: `full_text_search`
- retention:
  - lore entries are retained until archived or superseded;
  - provenance events may compact into summaries after a policy window;
  - deleted entries become tombstones until export/compaction.

Repository contracts:

- `create_world`
- `upsert_lore_entry`
- `supersede_lore_entry`
- `search_lore`
- `list_lore_by_entity`
- `record_lore_provenance`
- `export_lore_world`
- `import_lore_world`

Physical tables are generated as:

- `module_roleplay_lore_worlds`
- `module_roleplay_lore_entities`
- `module_roleplay_lore_lore_entries`
- `module_roleplay_lore_relationships`
- `module_roleplay_lore_timelines`
- `module_roleplay_lore_provenance_events`

Read-only query catalog examples:

- lore entries by world/entity/canon status;
- recently changed lore entries;
- provenance events for an entry;
- lore table growth by world.

TypeScript roleplay code can decide how lore enters prompts and how users review
candidate updates. Rust owns the stored record shape, revisions, search
repository, provenance writes, and import/export validation.

## Example: Simple Key/Value Module

Bundle:

- `module_id`: `simple_kv`
- `schema_version`: `1`
- store: `entries`
- capabilities:
  - required: `transactions`
  - optional: `json_documents`
- retention:
  - optional per-entry `expires_at`;
  - expired entries are removed by module maintenance.

Logical record:

- `scope_type`
- `scope_id`
- `key`
- `value_json`
- `revision`
- `created_at`
- `updated_at`
- `expires_at`

Repository contracts:

- `get_kv`
- `list_kv`
- `put_kv`
- `compare_and_swap_kv`
- `delete_kv`
- `expire_kv`

Physical table:

- `module_simple_kv_entries`

This module is intentionally boring and should be the first implementation
slice because it exercises descriptor validation, namespacing, migration state,
repository access, diagnostics, and retention without roleplay-specific
complexity.

## Diagnostics And Query Catalog

The storage diagnostics surface should include module registry data:

- registered modules and versions;
- installed versions;
- migration status;
- capability status;
- logical stores and physical table ownership;
- retention policy summaries;
- row counts and approximate size by module/store;
- degraded or blocked reasons.

The read-only query catalog should be able to list module-provided queries, but
execution still routes through Rust-owned repository/query functions. Catalog
entries are diagnostic conveniences, not a raw schema contract for agents.

## Minimal Implementation Slice

The first implementation should avoid roleplay-specific behavior and prove the
registry itself:

1. Add Rust descriptor types for module schema bundles, logical stores,
   capability requirements, and retention declarations.
2. Add a closed registry with one built-in `simple_kv` test/example module.
3. Persist installed module schema versions alongside core migrations.
4. Enforce namespace validation and blocked startup on invalid descriptors.
5. Add storage diagnostics projection for registered/installed modules.
6. Add a tiny repository and query catalog entry for `simple_kv`.
7. Add tests for valid registration, invalid names, capability gating, migration
   state, diagnostics, and SQLite-backed repository round trips.

Roleplay lore should come after this slice, using the same registry rather than
creating an independent lore storage path.

## Consequences

Positive:

- Future modules can keep data in the Crew DB without schema drift.
- SQLite remains simple while PostgreSQL remains feasible.
- Agents and Rusty View get safe visibility through diagnostics/query catalog
  surfaces.
- Roleplay lore can be specific in shape without becoming an external service
  or TS-owned schema island.

Costs:

- Adding a durable module requires Rust work, even if most module logic lives in
  TypeScript.
- Schema descriptors must stay disciplined enough to avoid becoming a second
  ORM.
- Export/import and rollback discipline must be designed before modules hold
  user-critical data.

## Deferred Decisions

- Exact PostgreSQL crate/pool choice.
- Whether descriptors later move to manifest/codegen input.
- Whether module repositories live directly in `core-persistence` or a sibling
  Rust crate consumed by it.
- How much arbitrary module lifecycle UI belongs in Rusty View.
- Whether production rollback hooks are ever supported beyond backup/restore.
