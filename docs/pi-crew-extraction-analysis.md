# Can the upstream code be extracted from pi-fleet?

Following up on the dependency audit: I traced the actual transitive-import closure of the 5 runtime symbols pi-crew uses (`Agent`, `streamSimple`, `getModels`, `getProviders`, `createAssistantMessageEventStream`) and built a tier table. The headline finding: **yes, extraction is feasible, but the cost isn't 4 functions — it's 9 files and one 16,136-LOC auto-generated monster that drags in everything by side effect**.

Implementation note: this analysis used the historical local audit checkout.
For new TypeScript brain-island work, use the current upstream source at
`https://github.com/earendil-works/pi`; do not treat older local paths or older
version pins as intentional implementation choices.

## Tier table (5 symbols → minimum vendored upstream)

| Tier | What's included | Files | LOC | External npm deps |
|------|-----------------|-------|-----|-------------------|
| **T0** | The 4 leaf files where the 5 symbols are *defined* | 4 | 715 | 0 |
| **T1** | T0 + the implementation files `Agent` needs (to actually run) | 9 | ~2,000 | 0 |
| **T2** | T1 + `models.generated.ts` (eagerly imported by `getModels`) | 10 | **~18,200** | 0 |
| **T3** | Full re-export closure (what `index.ts` of each package points at) | 66 | **~34,000** | 25 (most lazy) |

The **honest extraction cost is T2: ~18,200 LOC, 0 external runtime deps**. Anything less means writing a new agent loop from scratch, which is the "rewrite pi-agent-core" path.

## The hidden cost nobody sees in the import statements

`pi-crew/pi-service/src/workers/agent-worker-executor.ts` does:
```ts
import { Agent } from "@earendil-works/pi-agent-core";
import { streamSimple, getModels, getProviders } from "@earendil-works/pi-ai";
```

That looks like 4 functions. It's not.

**`getModels()` pulls `models.generated.ts` at module load.** This is a 16,136-LOC, 423 KB, single auto-generated file (per the header comment: *"This file is auto-generated by scripts/generate-models.ts. Do not edit manually"*). It contains the complete pricing/context-window/feature table for every model in every provider upstream supports — Bedrock, Anthropic, OpenAI, Mistral, Google, Vertex, Azure, GitHub Copilot, Cloudflare, etc. The exact line that does this is in `ai/src/models.ts:5-12`:

```ts
import { MODELS } from "./models.generated.ts";
const modelRegistry: Map<string, Map<string, Model<Api>>> = new Map();
for (const [provider, models] of Object.entries(MODELS)) {
  // build the registry on module load
}
```

So any worker that imports `getModels` (and all 3 production call sites do) is paying for ~16 KLOC of generated data — even if the worker only ever talks to one provider.

**`streamSimple()` pulls all 11 provider registrations at module load** via a side-effect import at the top of `stream.ts:1`: `import "./providers/register-builtins.ts";`. But — and this is the saving grace — the provider **implementations** are loaded lazily via `importNodeOnlyProvider()` dynamic calls. The only non-lazy thing in `register-builtins.ts` is registering factory functions; the actual SDK code (`@aws-sdk/client-bedrock-runtime`, `@anthropic-ai/sdk`, `@google/genai`, `@mistralai/mistralai`, `openai`) is only loaded when a worker actually calls a model on that provider.

**`Agent` itself is a 391-line class** that wraps `agent-loop.ts` (664 LOC) and `types.ts` (387 LOC). All three have **no top-level side effects** — they export things and define types, that's it. So `Agent` ships as pure code, no hidden load cost.

## What T2 looks like to vendor

```
pi-fleet/
└── vendor/
    └── upstream-pi-core/
        ├── agent/
        │   ├── agent.ts            (494 LOC — public class Agent)
        │   ├── agent-loop.ts       (664 LOC — runAgentLoop, runAgentLoopContinue)
        │   └── types.ts            (387 LOC — AgentContext, AgentEvent, AgentMessage, etc.)
        └── ai/
            ├── stream.ts            (65 LOC — streamSimple)
            ├── models.ts            (78 LOC — getModels, getProviders, calculateCost)
            ├── api-registry.ts      (85 LOC — registerApiProvider)
            ├── env-api-keys.ts     (187 LOC — getEnvApiKey)
            ├── utils/
            │   └── event-stream.ts  (78 LOC — createAssistantMessageEventStream, EventStream class)
            └── models.generated.ts (16,136 LOC — the auto-generated provider/model data)
```

Total: 9 hand-written source files + 1 generated data file = ~18,200 LOC. Zero external runtime deps (no `@anthropic-ai/sdk`, no `@aws-sdk`, no `typebox` — those are only needed by code that pi-crew never reaches).

## What T3 would look like (and why you don't want it)

If instead of tracing from the 5 symbols pi-crew uses, you trace from `index.ts` of each upstream package, the closure explodes:

- 66 files, ~34,000 LOC
- 25 external npm deps, including heavyweights: `@aws-sdk/client-bedrock-runtime`, `@anthropic-ai/sdk`, `@google/genai`, `@mistralai/mistralai`, `openai`, `http-proxy-agent`, `@smithy/*`, `typebox`
- Whole subdirs that pi-crew doesn't touch: `agent/src/harness/` (compaction, skills, session JSONL repos, system-prompt composition, prompt-templates), `agent/src/proxy.ts`, `ai/src/image-models*`, `ai/src/providers/{anthropic,bedrock,google,mistral,...}.ts` (1,000+ LOC each)

This is the **whole** upstream monorepo minus tests. Vendoring T3 would mean taking on the entire upstream maintenance surface — every provider upgrade, every new model, every API breaking change.

## What you give up by not vendoring at all (staying on npm)

- **Patch latency**: when a provider breaks a wire format, you have to wait for upstream to publish, or carry a patched fork in `dist/`.
- **Trust surface**: you import 16,136 LOC of generated data from an external package at module load. If a poisoned `models.generated.ts` shipped, every pi-crew worker would execute it.
- **Coupling to versioning**: the audited pi-fleet snapshot pinned an older package range. Current implementation should resolve `@earendil-works/pi-*` from `https://github.com/earendil-works/pi` rather than treating that old range as a design constraint.
- **Upstream's license terms**: it's MIT, so vendoring is legally clean if you preserve attribution.

## Recommendation: stay on the npm dependency, but trim T3 to T2 in your mental model

The pragmatic answer is: **don't vendor**. The cost of staying on the npm dep is:
- 1 package source decision for the current `https://github.com/earendil-works/pi` packages
- ~16 KLOC of generated data on every worker cold start
- Whatever surface area the 51 type-only imports expose to upstream renames

The cost of vendoring is:
- ~2 KLOC of hand-written code to maintain (`Agent`, `agent-loop.ts`, `streamSimple`, `getModels`, etc.)
- ~16 KLOC of generated data to regenerate when providers change pricing/models
- Your own CI step to regenerate `models.generated.ts` from upstream (or maintain it by hand — neither is great)

The break-even is roughly: if you find yourself patching upstream more than ~2x per year for things pi-fleet actually uses, vendoring starts to pay off. Until then, the npm dep is cheaper to maintain.

### Specific things worth following up on

1. **The 16,136-LOC generated file is the actual tax.** If the team ever wants to lighten the npm dependency, **trimming `models.generated.ts` to only the providers pi-fleet actually supports** (Anthropic, OpenAI — based on the runtime call sites) would drop T2 from 18,200 to ~2,200 LOC. The cost is losing auto-updates when upstream adds new models — you'd have to maintain your own generated list.
2. **The 51 type-only imports are the actual coupling risk.** They're cheap now (compile-time only), but every upstream rename of `AgentMessage`/`AgentEvent`/`AgentTool` will break the build. If you ever do vendor, you can drop these and use the structural mirrors in `guarded-tool-types.ts` everywhere.
3. **`Agent`'s implementation is genuinely small.** 391 lines of class body + 664 lines of loop logic + 387 lines of types. If you ever do want to escape upstream, writing your own `Agent`-equivalent is a 1-2 day project, not a 1-2 month one. The hard part is `models.generated.ts` (or replacing it) and the provider SDK integrations.
4. **No external SDKs in T2.** Vendoring the 9 hand-written files doesn't pull in `@aws-sdk`, `@anthropic-ai/sdk`, `@google/genai`, etc. — those are loaded by the provider modules that `register-builtins.ts` lazy-imports, and `register-builtins.ts` only matters if you actually call a model on that provider. A worker that only ever calls Anthropic would only load `@anthropic-ai/sdk` at first `streamSimple` call.

## TL;DR

> Is it possible to extract the upstream code we actually use?

Yes. The hand-written surface is 9 files, ~2,000 LOC, no external deps. **The hidden cost is `models.generated.ts` (16,136 LOC of generated provider/model data), which `getModels` eagerly pulls in.** Total realistic extraction cost: ~18,200 LOC. Anything bigger than that means you're re-vendoring upstream's whole surface, which is what the npm dep is for.

The code *is* extractable. The question is whether the 16 KLOC of generated data is worth re-maintaining, and that's a call based on how often you need to patch upstream vs how much friction the version pin causes. My read of the actual call sites: you almost never need to patch it, so the npm dep is the right answer.
